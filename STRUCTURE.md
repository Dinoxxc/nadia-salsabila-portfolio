## File Organization

### Pages (src/pages/)
- **index.html** - Portfolio home page with projects showcase
- **about.html** - About section with bio and introduction
- **education.html** - Education timeline and qualifications
- **experience.html** - Work experience and career history
- **projects.html** - 16 portfolio projects with modal details
  - Includes: ExpertConnect, LiveTips, CeremonyHub, Digital Invite, TrackFleet, LearnHub, ForecastAI, FreelanceHub, SmartCity, FactoryPro, GamePal, ConnectSphere, CryptoVault, TokenBridge, SmartHub, KoperasiHub, StoreHub, EscrowChain
- **articles.html** - 3 published research articles with modals
  - Includes: Hybrid ML, SOA Comparison, COCO-SSD
- **blog.html** - Legacy page (redirects to articles)
- **contact.html** - Contact form and information

### Assets (src/assets/)
- **images/** - Logo and profile images
- **styles/** - CSS files (can be extracted from HTML)
- **scripts/** - JavaScript files (can be extracted from HTML)

### Build Output (dist/)
- Auto-generated by Vercel during deployment
- All src/ files processed and output here
- Vercel.json points outputDirectory to dist/

## Routing

Via `vercel.json` rewrites:

```
/             → dist/index.html
/about        → dist/about.html
/education    → dist/education.html
/experience   → dist/experience.html
/projects     → dist/projects.html
/articles     → dist/articles.html
/blog         → dist/articles.html (backward compatibility)
/contact      → dist/contact.html
/cv           → dist/cv.html
```

## Development

### Building Locally
```bash

python -m http.server 8000
# or
npx http-server
```

### Deploying to Vercel
```bash
git add -A
git commit -m "commit message"
git push origin master
# Vercel auto-deploys on push
```

## Migration from static-site/

**Previous Structure:**
```
static-site/
├── *.html (8 files)
├── images/ (4 files)
└── *.pdf
```

**New Structure:**
```
src/
├── pages/ (8 files)
├── assets/
│   ├── images/ (4 files)
│   └── *.pdf
```

**Why?**
- Industry-standard structure
- Scalability for future components
- Clear separation of concerns
- Easier to add build processes later
- Better organized for maintenance
